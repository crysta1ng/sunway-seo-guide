1. Remove the toxic backlinks CRITICAL

The problem: ~230 of the site's 336 backlinks come from a spam link farm using anchor text like telegram @seo_linkk – mass backlinking and telegram @seo_anomaly - seo backlinks, black-links. This is a negative-SEO attack and can drag down rankings. It is still adding links.
Note: This is not a code change — it's done in Google Search Console. You do not need to edit the website. But it is the single most important action, so it's first.
The file is ready — you don't build anything. A complete, correctly-formatted disavow file is supplied with this guide: disavow_stories_sunway_com_my.txt (175 spam domains — the full "seo-anomaly" PBN network plus the compromised referring domains). Just upload it.

Steps

  1. Go to the Google Disavow Tool: https://search.google.com/search-console/disavow-links
  2. Select the property stories.sunway.com.my (or the sunway.com.my domain property — see note below).
  3. Upload disavow_stories_sunway_com_my.txt and confirm. Google reprocesses these links over the following weeks.
Property note: Disavow applies at the property level. If Search Console is set up as a domain property for sunway.com.my, the disavow covers all subdomains — in that case merge these 175 lines into any existing disavow file for that property rather than replacing it. If stories.sunway.com.my is its own URL-prefix property, upload the file as-is there.
Done when: the disavow file is uploaded in GSC and a copy is saved in your records.

2. Add structured data (JSON-LD schema) MEDIUM

Confirmed missing. The Semrush Site Audit crawl found schema.org / JSON-LD markup on 0 of 563 pages (Open Graph and Twitter cards are present on 542 — those are separate). So the site has no Article or Organization structured data and this needs to be added. If an SEO plugin (Yoast / Rank Math) is installed, its schema output is likely switched off — enabling it is the fastest fix.
Why it matters: structured data helps Google show rich results and makes articles easier for AI/answer engines to cite. Effort: low. After implementing, validate with Google's Rich Results Test (https://search.google.com/test/rich-results).

Option A — via SEO plugin (recommended, no code)

  • Yoast SEO: schema is output automatically. Ensure Settings → Site basics → Site representation is set to Organization = "Sunway Stories" with a logo. Per post, set the Schema tab (in the Yoast metabox) → Page type: Article.
  • Rank Math: Rank Math → Titles & Meta → Posts → set Schema Type = Article as the default. Set the org under Rank Math → General Settings → Knowledge Graph = Company.

Option B — manual (if no plugin handles it)

Add this to the article template (single.php) inside <head>, or via a hook in functions.php. Populate the fields dynamically from the post:

<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "<?php echo esc_js(get_the_title()); ?>",
  "image": "<?php echo esc_url(get_the_post_thumbnail_url(null,'full')); ?>",
  "datePublished": "<?php echo get_the_date('c'); ?>",
  "dateModified": "<?php echo get_the_modified_date('c'); ?>",
  "author": { "@type": "Organization", "name": "Sunway Stories" },
  "publisher": {
    "@type": "Organization",
    "name": "Sunway Stories",
    "logo": {
      "@type": "ImageObject",
      "url": "https://stories.sunway.com.my/path/to/logo.png"
    }
  },
  "mainEntityOfPage": "<?php echo esc_url(get_permalink()); ?>"
}
</script>

Site-wide Organization schema (add once, e.g. on the homepage template):

<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Organization",
  "name": "Sunway Stories",
  "url": "https://stories.sunway.com.my/",
  "logo": "https://stories.sunway.com.my/path/to/logo.png",
  "parentOrganization": { "@type": "Organization", "name": "Sunway Group" }
}
</script>
Validate: paste a live article URL into https://validator.schema.org and Google's Rich Results Test (https://search.google.com/test/rich-results). Both should report a valid Article with no errors. Do not run Option A and B together — that creates duplicate schema.

4. Strengthen internal linking HIGH

The problem: one article ("Pinnacle of Excellence") drives ~55% of all organic traffic and the top 3 pages ~80%. That page holds most of the site's ranking power but barely passes it to other articles. If it ever drops, traffic collapses.
Fix: add contextual internal links from the high-traffic pages to related lower-traffic articles, using descriptive anchor text. This spreads authority and lifts the weaker pages.
  1. In the Pinnacle and Brand-Evolution articles, add 3–5 in-body links to related stories (property, sustainability, township pieces) with keyword-rich anchors — e.g. Sunway City Iskandar Puteri, not "click here".
  2. Add a "Related Stories" block to the article template that surfaces same-category posts (most WordPress themes/plugins support this; verify it renders real <a href> links in the HTML, not JavaScript-only).
  3. Ensure category hubs (/sustainability, /innovation, /business, /society, /healthcare) link down to their best articles and each article links back up to its category.

Structural issues from the Site Audit to fix alongside

  • Fix the redirect loop first — it cascades. The page /the-joys-of-eating-at-the-banjaran/ is stuck in a 302 loop (it redirects to itself). That single broken page is the target of all 6 redirect chains and is the "incorrect page" flagged in the sitemap (#6). Give it a correct 200 response or a single clean 301 to the right destination, then the 6 linking articles resolve. The 6 source pages are listed in the workbook sheet "Redirect loops (6)".
  • ~3,968 internal links point to redirected URLs. Most trace to a handful of repeatedly-linked targets (like the loop above and old slugs). After fixing the loop, do a database search-replace of the old URLs to their final destinations across posts to clear the bulk.
  • 28 orphaned pages in the sitemap have no internal links pointing to them, and 30 pages have only one internal link. Add contextual links so every page you want indexed is reachable.
  • 254 pages sit more than 3 clicks deep. Surface important buried articles via category hubs, related-post blocks, or a "popular stories" module to shorten click depth.
Done when: the Banjaran page returns 200, every priority article has ≥3 inbound internal links, and top pages link out to ≥5 related articles.

No 404s, no accidental noindex, no redirect or non-critical issues — technically clean. Not-indexed has been falling (427 → 205 since April), so the trend is healthy. 8. Stop fbclid tracking URLs being indexed MEDIUM

Why: when links are shared on Facebook, a ?fbclid=… parameter is appended (e.g. /article/?fbclid=AbC123). Google can crawl these as separate URLs, creating duplicate versions of every page. This very likely feeds the 8 "Duplicate, Google chose different canonical" pages and some of the 191 "Crawled – not indexed" pages seen in GSC.

Primary fix — canonical tag (do this first)

Ensure every page outputs a self-referencing canonical to the clean URL without query parameters. The homepage already does this correctly (<link rel="canonical" href="https://stories.sunway.com.my/">). Confirm article templates do the same — Yoast and Rank Math handle this automatically, so verify rather than rebuild. With a correct canonical, Google consolidates the fbclid versions into the clean URL on its own.

Use robots.txt with caution. Blocking fbclid in robots.txt stops crawling but not indexing — and worse, if a tracking URL is already indexed, blocking it prevents Google from seeing the canonical, so it can get stranded in the index. Prefer the canonical fix above. Only add a robots.txt rule if the parameter URLs are not yet indexed and you want to save crawl budget:

# robots.txt — optional, only if these URLs are not already indexed
User-agent: *
Disallow: /*?fbclid=
Disallow: /*&fbclid=

Verify

  • In GSC, use URL Inspection on a ?fbclid= URL — it should report the clean URL as the "Google-selected canonical".
  • Re-check the "Duplicate, Google chose different canonical" bucket over the following weeks; it should shrink.

9. Address thin / low-value content HIGH
